# CLI/GUI Stdin Mode Separation

## Problem Statement

The application crashes immediately on startup when run as a windowed PyInstaller EXE (double-click), because Rich Prompt.ask() requires sys.stdin which is unavailable in GUI mode:

`
RuntimeError: input(): lost sys.stdin
  at rich.prompt.Prompt.ask()
  at input_handler.get_director_input()
  at main_orchestrator._process_single_director()
`

The root cause is that the app mixes interactive CLI input and batch processing in the same execution path, and the --windowed PyInstaller flag strips sys.stdin from GUI applications.

## Solution

Explicitly separate execution modes so that batch mode (the production default) has zero stdin dependency, and interactive mode is only activated via explicit CLI flags and only when a terminal is available.

## User Stories

1. As a director/processor, I want to double-click the EXE and have it run in batch mode automatically, so that the tool works without any command-line interaction.
2. As a director/processor, I want the EXE to fail gracefully with a clear message if no license key is stored and no terminal is available, so that I understand what to do next.
3. As an operator running the EXE from a command line, I want to pass --interactive and be prompted for director details, so that I can process emails one at a time interactively.
4. As an operator who needs to run the EXE from a terminal, I want a --console flag that forces CLI behavior even when the EXE is built as windowed, so that interactive mode works from cmd/powershell.
5. As an operator, I want --batch to explicitly invoke batch mode from the command line, so that the mode selection is unambiguous and not dependent on config files.
6. As a CI/automated system, I want batch mode to have zero stdin dependencies, so that the EXE runs correctly under any execution context (Task Scheduler, triggered by file watcher, etc.).
7. As a developer, I want the application to raise a clear RuntimeError if an interactive prompt is accidentally called during batch execution, so that bugs are caught immediately rather than silently ignored.
8. As a developer, I want all stdin-dependent code paths to be isolated in InputHandler and LicenseValidator, so that the rest of the application can be tested without mocking stdin.
9. As an operator, I want to see a help message listing available flags when I pass --help, so that I understand the execution modes.
10. As a license administrator, I want the application to prompt for a license key only when a terminal is available, so that the batch EXE does not crash on license entry.

## Implementation Decisions

### 1. Add rgparse to main_orchestrator.py

Replace the current un() logic (which reads un_mode from config) with CLI-driven mode selection:

- --batch (default) — fully automated, no stdin, reads from config directors
- --interactive — Rich prompts for director input, requires terminal
- --console — forces CLI behavior, same as --interactive but for EXE builds
- --help — standard argparse help listing all modes

CLI flags override the config un_mode setting. The un_mode config key is deprecated but still respected if no CLI flags are provided.

### 2. Create src/stdin_guard.py — new module

A small, isolated utility module providing:

- stdin_available() -> bool — returns sys.stdin is not None and hasattr(sys.stdin, 'isatty') and sys.stdin.isatty()
- equire_stdin(operation: str) -> None — raises RuntimeError(f"Interactive {operation} called outside terminal. Use --interactive flag from command line.") if stdin unavailable

This module has no dependencies on any other application module. It is the single choke-point for all stdin availability checks.

### 3. Guard InputHandler in src/input_handler.py

- Add rom src.stdin_guard import stdin_available, require_stdin
- get_director_input() — calls equire_stdin("prompt") at the top before any Prompt.ask()
- prompt_continue() — calls equire_stdin("continue prompt") at the top before Confirm.ask()
- stdin_available() is exported as a static method on InputHandler for convenience

### 4. Guard LicenseValidator in src/license_validator.py

- prompt_and_validate() — checks stdin_available() before attempting input(). If stdin unavailable and no stored key, writes a user-friendly message to stderr and returns None instead of crashing.
- On RuntimeError from stdin guard in main_orchestrator._validate_license(), the app prints a clear message and exits with code 1.

### 5. Guard MainOrchestrator._run_interactive() in src/main_orchestrator.py

- _run_interactive() is only called when --interactive or --console is passed OR when stdin_available() is True. Otherwise, _run_batch() is forced and a warning is logged.
- _validate_license() — if no stored key and stdin is unavailable, print message and exit. If stdin available, call prompt_and_validate().

### 6. ProgressManager — no changes required

Rich Console.print() writes to stdout (not stdin). All progress output methods are safe in windowed mode. No changes needed.

### 7. Batch mode has no stdin dependencies

After the above changes, the entire _run_batch() path, _run_interactive() path (guarded), _process_batch_director(), _process_director_by_name(), and _connect_to_outlook() paths all have zero stdin calls.

### 8. Build script — no changes required

uild.ps1 keeps --windowed. The --interactive and --console flags work from command line. Batch mode (default) works from double-click.

## Testing Decisions

### What makes a good test

Only test external behavior — that stdin availability is correctly detected, that guarded methods raise/return appropriately when stdin is absent, and that the argument parser correctly routes to batch or interactive mode. Do NOT test implementation details.

### Modules to test

1. **src/stdin_guard.py** (new): stdin_available() returns False when stdin is None or not a TTY; returns True when stdin is a TTY; equire_stdin() raises RuntimeError with expected message when stdin unavailable; does NOT raise when stdin is available.

2. **src/input_handler.py** — update existing tests: get_director_input() raises RuntimeError when stdin unavailable (new test); prompt_continue() raises RuntimeError when stdin unavailable (new test); existing tests with patch('builtins.input') still pass.

3. **src/license_validator.py** — update existing tests: prompt_and_validate() returns None and writes to stderr when stdin unavailable (new test); existing test still passes.

4. **src/main_orchestrator.py** — update existing tests: orchestrator routes to batch when --batch passed; routes to interactive when --interactive passed; forces batch mode when --batch passed even if config says interactive.

### Prior art

- 	est_input_handler.py uses patch('builtins.input') with side_effect — apply same pattern for stdin guard tests
- 	est_config_manager.py uses monkeypatch to isolate _get_appdata_dir — similar approach with patch('sys.stdin') for stdin guard
- 	est_main_orchestrator.py uses patch at module level for all COM/PDF dependencies — extend for argparse

## Out of Scope

- Removing --windowed from the build (keeping windowed as the production default)
- Converting the EXE to always show a console window
- GUI-based input alternatives (tkinter message boxes)
- Changing the Rich output layer (ProgressManager has no stdin dependencies)
- Modifying the batch director configuration format
- Adding a config option to override CLI flag behavior
- UPX compression or further bundle size reduction

## Further Notes

The Playwright browser path fix (copying chromium_headless_shell-1217 to _internal/playwright-browsers/ post-build) was completed in the same session as this issue. It is a prerequisite — without it, the app fails immediately at browser launch even before reaching stdin-related code.
